## D-001 ┬Ę ORRERY is a standalone repo, separate from the theory
Date: 2026-07-05 ┬Ę Status: Active
**Context:** The instrument (sims) and the science (the final theory) evolve on different clocks. **Decision:** ORRERY is its own repo/folder (`C:\ORRERY`), public on GitHub, versioned independently. **Rationale:** It is a *reusable general instrument*; coupling it into the theory repo would entangle two lifecycles and defeat the compounding. The science depends on ORRERY only through tool contracts. **Alternatives rejected:** live inside the theory repo (entangles versioning); a monorepo (premature). **Consequences:** the science imports nothing from ORRERY; it shells out to tool `.exe`s and reads declared output.

## D-002 ┬Ę `someone` is the first tool and the template

## D-011 ┬Ę Complexity ladder is CUMULATIVE (L0 base ┬Ę L1 +predators ┬Ę L2 +moving-lights ┬Ę L3 +night)
Date: 2026-07-05 ┬Ę Status: Active
**Context:** the contract labels levels "L0 none ┬Ę L1 predators ┬Ę L2 moving-lights ┬Ę L3 full (predators+night+food+moving)". The exact feature gating is an implementation choice within those labels. **Decision:** treat the ladder as **cumulative feature additions**: L0 = base survival economy (static lights + food zones + energy depletion; no predators/moving/night); L1 = L0 + predators; L2 = L1 + moving lights; L3 = L2 + night (= full). Food/energy is always on (the base stake at every level, so the zombie-vs-normal comparison is meaningful at all levels). **Rationale:** this makes **L2ŌåöL3 differ only by the night toggle** ŌĆö the clean A/B the science relies on (D-DAK-RNG, established.md F6 note), which requires L3 = L2 + night. It matches the contract's L3="full" and round-01's regime labels (the feature *added* at each level: L1 predators, L2 moving-lights, L3 night). **Alternatives rejected:** feature-isolated levels (L2 = moving-lights-only, no predators) ŌĆö breaks the L2ŌåöL3 night A/B and contradicts L3 containing predators. **Consequences:** documented in MODULE.md; the selftest's fair-layout assertion holds across all four cumulative levels.

## D-012 ┬Ę RNG architecture: host mt19937_64 + stateless counter Gaussian + purpose-keyed splitmix64 (no curand)
Date: 2026-07-05 ┬Ę Status: Active
**Context:** the prototype's per-neuron noise loads one agent-level `curandState` into every thread, each thread draws independently, and all threads write the state back to one global slot (lines 185/393/507) ŌĆö a last-writer race AND identical per-neuron noise; plus wall-clock seeds (lines 545, 590). **Decision:** drop device curand entirely. Use (i) a seeded host `std::mt19937_64` for weight init / zombie assignment / evolution (replica r seeded seed+r, fixed draw order), (ii) a **stateless counter-based Gaussian** (splitmix64 hash ŌåÆ two uniforms ŌåÆ BoxŌĆōMuller) for per-step neuron noise keyed by (seed+replica, agentId, neuronId, step) ŌĆö no device RNG state, hence no writeback race and no memory, fully deterministic, and (iii) a purpose-keyed splitmix64 for env layout (see D-DAK-RNG fix). **Rationale:** removes the race *by construction* (there is no shared RNG state to race on) and makes every random value a pure function of its coordinates, which is the strongest form of determinism and trivially reproducible/inspectable. **Alternatives rejected:** per-neuron persistent `curandState` arrays (works, but adds ~48 B/neuron of state and an init kernel, and still relies on curand's cross-version stability); serial thread-0 noise draws (deterministic but serializes N draws, slowing the hot loop). **Consequences:** determinism is a pure-arithmetic guarantee; documented in MODULE.md. Kernel FP reductions stay fixed-order tree reductions (already deterministic); host ensemble aggregation uses index-order Kahan.

## D-014 ┬Ę `someone` golden runtime is ~8 min (bandwidth-bound); keep fp32; fp16 is a future golden-superseding option
Date: 2026-07-05 ┬Ę Status: Active
**Context:** the contract-fixed golden config (`--pop 200 --gens 200 --steps 800 --N 256 --ensemble 4`) is dominated by the per-step weight matvecs (recurrent `W` N├ŚN and predictor `P` N├ŚF, FŌēł2N ŌĆö the self-model re-reads ~900 KB of weights **per agent per step**). This exceeds ARCHITECTURE ┬¦11's "golden suite < 5 min" NFR. Experiments confirmed it is genuinely bandwidth-bound, not occupancy- or barrier-bound: pop 200├Śens 4 (200 blocks) and pop 800├Śens 1 (800 blocks) take the same time (more blocks don't help), and replacing tree reductions with warp-shuffle reductions and parallelizing the thread-0 motor matvec gave no measurable change. (b) shrinking the golden config ŌĆö a contract MAJOR change, operator-gated, rejected. **Consequences:** `--golden` (and CI) takes ~8 min for this one tool; the NFR is knowingly missed with rationale. The **science's real experiment (S5) uses a lighter but statistically stronger config** (smaller N/pop/gens, but nŌēź20 replicas) that runs in minutes ŌĆö heavy statistics matter more than a heavy single episode (BITE #3). Determinism verification runs `--golden` 3├Ś (~25 min, one-time).

## D-013 ┬Ę Golden hash domain = canonical JSON of {seed, params, result, gates, verdict}
Date: 2026-07-05 ┬Ę Status: Active
**Decision:** the golden hash is `blake2b` over a canonical serialization of `{seed, params, result, gates, verdict}` (floats at fixed `%.6f`, keys in fixed declared order), **excluding** `tool`, `version`, and `notes`. **Rationale:** the golden must be *behavior*, not label ŌĆö a pure kernel rewrite that reproduces results (and may bump `version`) must keep the golden ("the golden is load-bearing, the code is ephemeral"). `notes`/timing are explicitly non-declared. Fixed `%.6f` gives run-to-run byte-stability (the computation is bit-deterministic) and a sane cross-rewrite tolerance. **Consequences:** `--golden` recomputes this exact canonical form; `goldens/someone/NOTE.md` documents the domain; the full JSON envelope still prints tool/version/notes for humans.
