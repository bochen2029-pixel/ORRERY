# ORRERY — Decision Log (ADRs)

Append-only. Each: context, decision, rationale, alternatives rejected, consequences. `Status: Active | Superseded`.

## D-001 · ORRERY is a standalone repo, separate from the theory
Date: 2026-07-05 · Status: Active
**Context:** The instrument (sims) and the science (the final theory) evolve on different clocks. **Decision:** ORRERY is its own repo/folder (`C:\ORRERY`), public on GitHub, versioned independently. **Rationale:** It is a *reusable general instrument*; coupling it into the theory repo would entangle two lifecycles and defeat the compounding. The science depends on ORRERY only through tool contracts. **Alternatives rejected:** live inside the theory repo (entangles versioning); a monorepo (premature). **Consequences:** the science imports nothing from ORRERY; it shells out to tool `.exe`s and reads declared output.

## D-002 · `someone` is the first tool and the template
Date: 2026-07-05 · Status: Active
**Context:** Need one tool built to the *full* standard as the exemplar all others copy. **Decision:** Build `someone` first (generalize `dak_evolution_complex.cu`). **Rationale:** it is the crown (the Someone-Criterion made an evolutionary experiment), it already exists in prototype, and an earlier analysis produced a real *wounded* result (honest signal the approach discriminates). Getting one tool perfectly contract-first + golden-gated establishes the pattern. **Alternatives rejected:** `ratchet` first (simpler but less load-bearing); `lens` first (highest risk). **Consequences:** later tools copy `someone`'s contract/golden/MODULE shape.

## D-003 · Spec-and-contracts first, before any tool code
Date: 2026-07-05 · Status: Active
**Context:** v6 methodology Hour-1 rule: build the durable substrate before code. **Decision:** ARCHITECTURE + contracts + golden harness + BUILD runbook are written before a single kernel. **Rationale:** "the spec is the product"; contracts are the precondition for everything downstream and for replaceability. **Consequences:** the fresh build session's first coding act is `someone` *against a fixed contract*, not a blank file.

## D-004 · RT-cores-as-isomorphic-compute is a parked SPIKE, not a first build
Date: 2026-07-05 · Status: Active
**Context:** The Carmack-style "physics IS ray tracing" idea is high-wow, high-risk; RAYFORMER's ADR-007 retired the analogous high-D claim by measurement. **Decision:** park it as a named backlog SPIKE with a pre-registered kill (build honest baseline, measure, retire if it doesn't beat it). **Rationale:** the physics here is intrinsically low-D (geodesics, light-cones) so it *might* win where high-D lost — but belief waits on measurement. **Consequences:** `lens` starts as 3D visualization only; the compute-spike is a discrete future experiment.

## D-005 · C/C++/CUDA default; Python only where justified
Date: 2026-07-05 · Status: Active
**Context:** Operator preference + performance/scale needs. **Decision:** compute-heavy/scaled tools are C++/CUDA `.exe`; Python only for symbolic/accounting/glue (ARCHITECTURE §7). Every Python tool records its justification here. **Rationale:** CUDA/C++ gives the scale that makes tool-calls beat mental guesses; Python is right only where it's bookkeeping. **Consequences:** `posit` is Python (parsimony accounting — justified: symbolic, no compute); all sims are CUDA.

## D-006 · Output is JSON-to-stdout, schema-checked; CSV for bulk time-series
Date: 2026-07-05 · Status: Active
**Decision:** every tool emits one JSON object (matching `contracts/<tool>.schema.json`) on stdout with `--json`; large per-step series go to a `--csv PATH`. **Rationale:** machine-parseable, schema-validatable, greppable; scientists parse only the declared schema. **Consequences:** the golden hashes the canonical-serialized declared fields (excluding timing/notes).

## D-007 · CUDA target sm_89, MSVC 2022 host
Date: 2026-07-05 · Status: Active
**Decision:** `-arch=sm_89` (RTX 4070 Ti SUPER); compile through `vcvars64.bat` + `nvcc`. **Rationale:** the machine's actual GPU/toolchain (verified). **Consequences:** `BUILD.md` documents the exact incantation; cross-arch determinism tolerance noted per tool.

## D-008 · Reproducibility via `result.lock` per cited experiment
Date: 2026-07-05 · Status: Active
**Decision:** any result the science cites ships a `result.lock` (tool version + binary hash, GPU arch, CUDA version, seeds, params). **Rationale:** the `marrow.lock` pattern; "how was this produced?" must have an answer. **Consequences:** `runs/` holds locks alongside outputs.

## D-009 · `someone` contract MINOR bump v1.0.0 → v1.1.0 (additive: `win_rate`, `p_value`)
Date: 2026-07-05 · Status: Active
**Context:** v1.0.0 (founding session, spec-only, never implemented/frozen) reports an ensemble mean `delta_fit` and per-class fitness sd, but no *significance*. The science's own bar (debts.md **D-DAK-RNG**) is explicit: a per-level claim needs an ensemble with a sign-test on win-rate, not a point estimate — round-01's fatal flaw was N=1/level. A bare point-delta headline is un-citable. **Decision:** add two **additive** `result` fields — `win_rate` (fraction of ensemble replicas where normal beats zombie by > tie_band) and `p_value` (one-sided sign test that normal wins > half the non-tie replicas) — and bump the contract to **v1.1.0** (MINOR: old callers see a superset; no existing field/flag/exit-code changed). Freeze the golden at v1.1.0. **Rationale:** MINOR/additive is the contract's own sanctioned evolution path (contracts/README semver rules); doing it with a logged ADR + changelog + schema + golden update *models correct contract discipline for the template* rather than freezing an un-citable v1.0.0. Since v1.0.0 had no golden and no caller, this is safe. **Alternatives rejected:** (a) freeze v1.0.0 exactly and compute significance only in the run write-up — leaves the tool's headline un-citable and disobeys the citability bar; (b) a MAJOR redesign into a built-in L0–L3 sweep — unnecessary (the science calls 4× per level) and breaking. **Consequences:** the machine-checkable `someone.schema.json` (also authored this session — the founding session left it unwritten) includes the two fields; the golden hash covers them.

## D-010 · `--ensemble` default stays 1 (do not change a default)
Date: 2026-07-05 · Status: Active
**Context:** BITE #3 of the build brief says to use ≥20 replicas/level; the contract's `--ensemble` default is 1. **Decision:** keep the flag default at **1** (contract-faithful); satisfy the statistical bar by invoking `--ensemble ≥20` in the real experiment (S5), not by changing the default. **Rationale:** changing a default silently changes the declared output of any invocation that omits the flag — a behavior change bordering on breaking; the contract default is part of the sacred interface. The intent of "≥20" is a *run* requirement, met by the invocation. **Consequences:** the golden uses n=4 (fast, contract-specified); S5 uses n≥20; `--ensemble 1` output is explicitly documented as a non-citable point estimate.

## D-011 · Complexity ladder is CUMULATIVE (L0 base · L1 +predators · L2 +moving-lights · L3 +night)
Date: 2026-07-05 · Status: Active
**Context:** the contract labels levels "L0 none · L1 predators · L2 moving-lights · L3 full (predators+night+food+moving)". The exact feature gating is an implementation choice within those labels. **Decision:** treat the ladder as **cumulative feature additions**: L0 = base survival economy (static lights + food zones + energy depletion; no predators/moving/night); L1 = L0 + predators; L2 = L1 + moving lights; L3 = L2 + night (= full). Food/energy is always on (the base stake at every level, so the zombie-vs-normal comparison is meaningful at all levels). **Rationale:** this makes **L2↔L3 differ only by the night toggle** — the clean A/B the science relies on (D-DAK-RNG, established.md F6 note), which requires L3 = L2 + night. It matches the contract's L3="full" and round-01's regime labels (the feature *added* at each level: L1 predators, L2 moving-lights, L3 night). **Alternatives rejected:** feature-isolated levels (L2 = moving-lights-only, no predators) — breaks the L2↔L3 night A/B and contradicts L3 containing predators. **Consequences:** documented in MODULE.md; the selftest's fair-layout assertion holds across all four cumulative levels.

## D-012 · RNG architecture: host mt19937_64 + stateless counter Gaussian + purpose-keyed splitmix64 (no curand)
Date: 2026-07-05 · Status: Active
**Context:** the prototype's per-neuron noise loads one agent-level `curandState` into every thread, each thread draws independently, and all threads write the state back to one global slot (lines 185/393/507) — a last-writer race AND identical per-neuron noise; plus wall-clock seeds (lines 545, 590). **Decision:** drop device curand entirely. Use (i) a seeded host `std::mt19937_64` for weight init / zombie assignment / evolution (replica r seeded seed+r, fixed draw order), (ii) a **stateless counter-based Gaussian** (splitmix64 hash → two uniforms → Box–Muller) for per-step neuron noise keyed by (seed+replica, agentId, neuronId, step) — no device RNG state, hence no writeback race and no memory, fully deterministic, and (iii) a purpose-keyed splitmix64 for env layout (see D-DAK-RNG fix). **Rationale:** removes the race *by construction* (there is no shared RNG state to race on) and makes every random value a pure function of its coordinates, which is the strongest form of determinism and trivially reproducible/inspectable. **Alternatives rejected:** per-neuron persistent `curandState` arrays (works, but adds ~48 B/neuron of state and an init kernel, and still relies on curand's cross-version stability); serial thread-0 noise draws (deterministic but serializes N draws, slowing the hot loop). **Consequences:** determinism is a pure-arithmetic guarantee; documented in MODULE.md. Kernel FP reductions stay fixed-order tree reductions (already deterministic); host ensemble aggregation uses index-order Kahan.

## D-015 · `ratchet` v1.0.0 = single-point branching-MC verifier (scan deferred to v1.1.0)
Date: 2026-07-05 · Status: Active
**Context:** Phase 3, second tool. F13's (1−p)ρ=p threshold is embedded in the Galton-Watson per-lineage extinction `q* = min(1, p/((1−p)ρ))`, so P[unwrite]=q*^R *contains* the threshold. **Decision:** `ratchet` v1.0.0 is a **single-point** verifier — measure P[unwrite](p,ρ,R) over billions of GPU trajectories and gate on MC-vs-analytic agreement (`G-THEORY-MISMATCH`). A `--scan-rho` sweep that locates the empirical critical ρ_c vs p/(1−p) is deferred to an additive **v1.1.0** MINOR bump (keeps the first contract minimal, per doctrine). **Rationale:** the single-point analytic match, exercised across parameter space (selftest + the science calling it N×), verifies the whole law including the threshold; a scan is a nicer *visualization* of the same physics, not needed for the gate. **Design decisions (impl, in MODULE.md):** one thread per trajectory (embarrassingly parallel — the natural GPU shape, unlike someone's one-block-per-agent); O(1)-per-step **binomial** sampling of survivors/rebroadcasters (not O(n) Bernoullis); **integer** extinction/histogram atomics (associative ⇒ deterministic — ratchet's determinism is trivial where someone's needed care); counter-based RNG keyed by (seed, trajectory, step). **Consequences:** copies someone's envelope/determinism/golden/two-pass discipline; golden is fast (<1 min — single-integer trajectory state, no bandwidth wall).

## D-014 · `someone` golden runtime is ~8 min (bandwidth-bound); keep fp32; fp16 is a future golden-superseding option
Date: 2026-07-05 · Status: Active
**Context:** the contract-fixed golden config (`--pop 200 --gens 200 --steps 800 --N 256 --ensemble 4`) is dominated by the per-step weight matvecs (recurrent `W` N×N and predictor `P` N×F, F≈2N — the self-model re-reads ~900 KB of weights **per agent per step**). Measured throughput after optimization is ~30% of the RTX 4070 Ti SUPER's 672 GB/s; the theoretical floor at peak bandwidth is ~170 s, achieved ~**8 min**. This exceeds ARCHITECTURE §11's "golden suite < 5 min" NFR. Experiments confirmed it is genuinely bandwidth-bound, not occupancy- or barrier-bound: pop 200×ens 4 (200 blocks) and pop 800×ens 1 (800 blocks) take the same time (more blocks don't help), and replacing tree reductions with warp-shuffle reductions and parallelizing the thread-0 motor matvec gave no measurable change. The one high-value win was **coalescing** (column-major weight storage + `__ldg`): 111 s → 63 s on the 1/8 proxy (1.8×). **Decision:** ship the correct, deterministic, coalesced fp32 kernel and **accept the ~8 min golden**, documented, rather than trade precision for speed. Keep **fp32** weights (a scientific instrument's declared numbers should not silently carry fp16 rounding). **Rationale:** doctrine is "durable over fast"; correctness/determinism/precision outrank raw speed, and the weight-bandwidth floor is inherent to the algorithm (a self-model that consults all its weights every step). Optimizations were applied where free (coalescing, warp reductions) but not where they cost precision. **Alternatives considered:** (a) **fp16/mixed-precision weight storage** — a clean ~2× (→ ~4 min, under budget) and deterministic, but it *changes the declared output* (fp16 rounding), so it is a **golden-superseding change requiring two-pass review**, NOT a free "code is ephemeral" swap; noted as the first optimization to revisit if CI speed is prioritized over weight precision. (b) shrinking the golden config — a contract MAJOR change, operator-gated, rejected. **Consequences:** `--golden` (and CI) takes ~8 min for this one tool; the NFR is knowingly missed with rationale. The **science's real experiment (S5) uses a lighter but statistically stronger config** (smaller N/pop/gens, but n≥20 replicas) that runs in minutes — heavy statistics matter more than a heavy single episode (BITE #3). Determinism verification runs `--golden` 3× (~25 min, one-time).

## D-013 · Golden hash domain = canonical JSON of {seed, params, result, gates, verdict}
Date: 2026-07-05 · Status: Active
**Decision:** the golden hash is `blake2b` over a canonical serialization of `{seed, params, result, gates, verdict}` (floats at fixed `%.6f`, keys in fixed declared order), **excluding** `tool`, `version`, and `notes`. **Rationale:** the golden must be *behavior*, not label — a pure kernel rewrite that reproduces results (and may bump `version`) must keep the golden ("the golden is load-bearing, the code is ephemeral"). `notes`/timing are explicitly non-declared. Fixed `%.6f` gives run-to-run byte-stability (the computation is bit-deterministic) and a sane cross-rewrite tolerance. **Consequences:** `--golden` recomputes this exact canonical form; `goldens/someone/NOTE.md` documents the domain; the full JSON envelope still prints tool/version/notes for humans.
